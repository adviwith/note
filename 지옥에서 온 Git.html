버전 관리 시스템에는 본질과 혁신이 있다고 생각하는데 이 강의에서는 본질부터 시작할 것입니다. 본질은 보편적이고 쉽지만, 혁신적인 요소는 아주 편리하지만 어렵습니다.

Git이라는 버전 컨트롤 시스템은 어렵고 복잡합니다. 혁신에 혁신을 거듭하면서 기능이 보태지는 과정에서 정말로 어려워졌습니다.

개발자들이 어려운 Git을 광범위하게 사용하는 이유 : 우리의 현실은 Git보다 훨씬 더 지옥같기 때문

복잡하지 않은 프로젝트에 복잡한 Git을 더하면 프로젝트의 복잡도는 과도해짐, 하지만 복잡한 프로젝트에 Git을 더하면 프로젝트의 복잡도가 오히려 낮아진다. 자신의 상황에 따라 적절한 수단을 선택해야 함.

버전이 무엇인가 : 모든 변화를 버전이라 하지 않음, 버전은 의미 있는 변화를 의미함. 작업이 완결된 상태가 버전 git을 처음 쓴다면, 이 버전이 자신이 작성한 버전이라는 것을 알리기 위해 이름을 세팅해야 함 git commit 하면 vim이 실행되는데 여기에 현재 버전의 메시지를 적는 것, 이 버전이 어떤 변화를 담고 있는지, 왜 변경되었는지. commit 메시지라고도 함

어떤 새로운 파일이 생겼을 때 그 파일에 대한 버전 관리를 git에게 명령할 때 add사용 이미 버전 관리가 되고 있는 파일이 수정되어 버전을 생성할 때에도 add 사용 -최초로 추적할 때에도 add사용, 최초로 파일이 추적될 떄도 add

commit 전에 add꼭 하기 : 지금까지 한 것이 제일 자주 사용하고 중요함. 모든 버전 관리시스템에서 똑같이 하는 것

왜 git이 add라는 과정을 포함하는 것일까. [add : commit 대기 상태로 만듦] 프로젝트를 하다 보면 여러 소스코드를 수정하게 된다. 그러다 보면 commit할 시기를 놓칠 때가 있다. commit하나는 하나의 작업을 담고 있는 것이 이상적이다. 그럴 때 많은 작업들을 하나의 버전에 담지 않고 git add를 통해서 원하는 파일들만 선택하여 버전을 만들 수 있다.

이전의 서브버전같은 경우는 한번에 해야 됐지만 git은 선택적으로 commit할 수 있다.

commit 대기 상태를 git은 stage area라고 부름 git add 하면 그 파일은 스테이지에 올라가고 그 상태로 commit하면 스테이지 위에 올라가 있는 파일들의 변경이 버전이 되는 것

stage외에 repository가 있는데 repository는 commit된 파일이 저장되는 곳

각각의 commit은 고유한 값이 있음 = 복잡하게 생긴 값 (SHA-1 해시값)

git diff는 마지막 commit이후의 변경점을 보여주는데, commit을 하기 전에 자기가 한 작업이 문제가 있는지 없는지를 마지막으로 리뷰할 수 있는 기회로 활용. 작업이 길어지면 어떤 작업을 했는지 기억을 잘 못하기 때문에 변경된 부분을 봄으로써 자신이 한 작업에 문제가 없는지 체크할 수 있음.

reset & revert

항상 우리는 경계를 햇갈려요, git reset이라 했을 때 해당 id로 돌아가는건지 그것도 지우는건지 햇갈린단 말이에요.(해당으로 돌아감) git은 보통 정보를 잘 삭제하지는 않아요. git reset해서 우리 눈에 안 보이더라도 되돌아갈수도 있어요. 버전을 공유한 뒤에는 절때 reset을 하면 안됩니다. reset ~ --hard외에도 soft등이 있는데, 나중에 원리를 살펴본 뒤 섬세하게 리셋을 살펴보기로 함. 일단은 hard를 쓰되 hard는 위험하다는 것을 알고 있어야 함.

revert는 커밋을 취소한 내용을 새로운 버전으로 만드는 명령인데, 일단은 이런게 있다는 것만 알고 있고 추후에 자세히 설명하도록 함

앞으로는 git이라는 구체적인 제품이 버전 관리 시스템에 가져온 여러가지 혁신들을 배울 것입니다. 그러므로 혁신은 나중에, 천천히 배우시면 됩니다.

우리가 버전 관리를 쓰는 가장 중요한 이유는 '벡업'입니다. 아직은 git이 가지고 있는 세련된 벡업에 대해 배우지 않았습니다. 만약 지금 잠시 수업을 중단한다면 여러분이 버전 관리하는 디렉토리를 클라우드 서비스에 올려서 백업하십시오. 이렇게 하면 혹시나 문제가 생겨도 완벽히 백업할 수 있습니다. 혼자서의 개발은 이걸로도 충분합니다. 협업할 때가 되면 이걸로는 택도 없습니다. 그때가 되면 중단했던 수업을 다시 시작하면 됩니다. (본인의 필요에 의해). 그때까지는 본인 스스로 수련하는 시간을 가졌으면 좋겠습니다.

만 개의 파일이 각각 10GB짜리의 엄청 큰 파일이더라도 같은 내용이라면 만 개의 파일은 하나의 오프젝트 파일을 가르킨다. 어마어마한 중복을 방지해 줌.

파일의 내용이 같다면 이세상의 누가 작성한 것이건 간에 이름이 같은 값이다. (해시값을 이용하는 듯)

 

  • object 파일명의 원리

git은 파일의 내용 기반으로 object 파일의 이름을 만든다.

"Hi"와 "Hello"는 누가 하던 간 같은 해시값을 가짐. git은 SHA1이라는 해시 알고리즘을 통해 파일의 이름을 만듦

git은 파일의 내용이 a라고 적혀있으면 a라는 정보와 몇 가지 부가적인 정보를 추가해서 압축합니다(SHA1을 통해 압축). 그 값에 해당하는 파일을 object에 저장하고, index라는 파일에 f1.txt.라는 파일은 이런 내용이다 라고 (해시값을)적어주면 끝나는 것입니다.

  • commit의 원리

여러분이 작성한 commit 메세지에 해당되는 정보가 objects라는 디렉토리 안에 저장되어 있습니다. 즉 여러분이 한 커밋(버전)도 마치 파일처럼 object내에 들어가는 것입니다. git에서는 commit도 내부적으로는 객체라고 할 수 있습니다

commit이 담긴 객체 내에는 tree라는 내용 옆에 object가 링크되어 있는데, 이 객체에는 현재 버전의 파일의 이름과 내용이 링크되어 있습니다.

f2.txt파일의 내용을 바꾸고 새로 commit을 하면 새 commit이라는 객체 내에 tree와 parent라는 object가 링크되어 있는데, parent는 이전 commit으로 링크됨,

commit에는 주요한 정보가 2개가 있습니다. 첫번쨰는 이전 커밋이 무엇인가 부모를 나타내는 parent라는 값이 있고요 두번째는 commit이 일어난 그 시점에 작업 디렉토리에 있는 파일의 이름과 그 파일의 이름이 담고 있는 내용 사이의 정보가 tree라는 곳에 담겨 있습니다. 그래서 각각의 버전마다 서로 다른 tree를 가리키고 있고 그 tree에는 파일의 이름과 그 파일에 담겨 있는 내용이 각각 링크가 되어 있기 때문에 우리는 버전에 적혀 있는 트리를 통해서 버전이 만들어 진 시점에 프로젝트 폴더에 대한 상태를 얻어낼 수 있습니다. 그러한 것을 '사진을 찍었다' 즉 스냅샷이라고 표현하는데, 각각의 버전은 버전이 만들어진 시점의 파일을 상태를 tree라는 스냅샷을 통해 보관하고 있는 것입니다.

각각의 commit은 tree를 가지고 있어서 commit이 이루어진 시점의 파일의 이름과 내용을 담고 있습니다.

디렉토리를 add해서 commit하면 commit 내의 tree에 들어 있는 tree(add한 디렉토리)에 파일의 내용이 담겨 있습니다.

object라는 디렉토리에 들어가는 파일이 objects라는 파일이고 objects파일은 세 가지 중 하나입니다. 하나는 파일의 내용을 담고 있는 blob이고 두 번째는 어떤 디렉토리의 파일명과 그 파일명에 해당되는 내용(blob)에 대한 정보를 담고 있는 tree입니다 세 번째, commit입니다. commit도 object id를 가지고 있습니다.

 

  • status의 원리

정확한 작동 방식은 모르겠지만. .git에는 index라는 파일이 있습니다. 아주 중요한 파일이죠. index안에 담긴 파일이 마지막 commit과 일치한다면 commit할 것이 없다는 의미 아닐까요? 파일 내의 내용을 바꾸면, 수정되었으며 commit에 포함되지는 않았다고 알려줍니다. (git status) git add를 하면 수정되었으며 commit에 포함되었다고 알려줍니다. index를 다시 보면 수정된 파일 내의 내용의 object가 담겨 있습니다. (마지막 commit의 내용과는 다름-수정했으니) 이를 통해 commit 대기 상태라는 것을 아는 것 아닐까

  • branch 정보확인

여기서는 branch 간의 차이점을 확인하는 방법을 설명합니다.

여러분이 branch를 만들면, 상당한 편리함과 동시에 상당한 복잡성을 가지게 됩니다. git이 복잡하다는 것은 이러한 편리함에서 오는 대가인 것 같아요.

이번 시간에는 branch를 만들었을 때 현재 어떤 상황인지를 판단할 수 있게 확인 방법을 배워보도록 하겠습니다.

xxxxxxxxxx
$ git log
$ git log --branches --decorate
(HEAD -> exp) 현재 체크아웃 된 branch는 exp
$ git log --branches --decorate --graph

log --branches는 모든 저장소에 있는 branch를 보여줍니다.

log --graph는 선으로 커밋을 따라 표시해줍니다. branch가 분기되면 각각 커밋되면 시각적으로 확인할 수 있습니다.

​x
$ git checkout master
$ vim graph.txt
$ vim add graph.txt
$ vim commit -m "5"
$ git log --branches --decorate --graph
​
$ git log --branches --decorate --graph --oneline #간결하게 표시

commandline을 사용하는 이유는 가치있기 때문입니다. 하지만 상황에 따라서 gui툴인 소스코드와 같은 다른 적합한 도구를 사용할 수도 있습니다. 하나만 고집할 필요는 없는 것 같아요.

xxxxxxxxxx
$ git log master..exp
$ git log exp..master

master에는 없고 exp에는 있는 것, exp에는 없고 master에는 있는 것을 비교해서 보여줍니다.

-p를 통해 각각의 버전에서 어떤 차이가 있는지 확인하실 수도 있습니다.

diff를 통해 각각의 branch의 현재 상태를 비교할 수 있습니다.

  • git 수련

이 수업에서는 지금까지 배운 git으로 수련을 하실 수 있도록 공식 홈페이지의 내용을 소개해드립니다. 실행은 직접 해보시길 바랄께요. 그리고 fast forward와 merge commit 의 차이점도 소개해드립니다.

홈페이지의 Documentation에서 Book이라는 페이지에서 Pro Git이라는 오픈 라이센스 책이자 git의 공식 메뉴얼로 git에 대해 공부하실 수 있습니다.

basic branching and merging을 보도록 하겠습니다.

이 문서를 보고 수련하시면 좋을 것 같습니다. 하지만 좀 어려운 것 같으니 제가 먼저 한번 설명해드리고 여러분이 수련하시면 좋을 것 같습니다.

어떠한 이슈가 생기면 git은 branch를 만듦니다. 

xxxxxxxxxx
$ git checkout -b issue53

branch만들기와 checkout동시에 하는법 '-b'

급하게 처리해야 될 일이 생겨서 다시 master branch에서 branch를 만들어야 하는 상황이 생겼습니다. 

xxxxxxxxxx
$ git checkout master
$ git checkout -b hotfix
#hotfix에서 수정
$ git checkout master
$ git merge hotfix
#Fast-forward라고 뜸

Fast-forward : merge를 했을 때 여러분이 경험할 수 있는 첫 번째 상황입니다.

master branch는 hotfix를 병합하기 전까지 어떠한 수정도 하지 않았습니다.

이 경우에는 master branch가 가리키는 commit을 hotfix가 가리키는 commit으로 '빨리감기'하면 병합이 끝납니다. 이 경우를 Fast-forward라고 합니다.

별도의 commit이 생성되지 않고 master가 가리키는 commit이 무엇인지를 바꾸기만 합니다(hotfix)

xxxxxxxxxx
$ git branch -d hotfix #hotfix 제거

다시 issue53로 돌아가서 issue를 해결하죠

xxxxxxxxxx
$ git checkout issue53
#issue해결 = 수정
$ git checkout master
$ git merge issue53

merge를 하면 "Merge made by the 'recursive' strategy"라고 나옵니다.

master의 내용이 분기된 후 수정되었기 때문에 fast-forward를 할 수 없습니다.

이 때 git은 내부적으로 master와 issue53의 공통 조상을 찾은 후 three-way merge라는 내부적인 방법을 이용해서 둘을 합치고 두개를 합쳤다 라는 것을 알려주는 별도의 commit을 만듭니다(자동으로).

2개의 부모로부터 비롯되었다는 것을 담고 있는 새로운 commit이 생겨납니다.

fast-forward가 아닌 merge는 merge commit이라는 commit을 생성합니다. fast-forward의 경우에 새 commit이 생성되지 않았던 것과 다르게 말이죠.

 

  • branch 병합 시 충돌해결

이번 시간에는 branch를 병합 할 때 git이 자동으로 처르해주는 소중한 작업이 무엇인가를 소개하고, 자동으로 병합할 수 없는 경우에는 어떻게 이를 수동으로 처리해야 하는지를 소개합니다.

merge를 하는 과정에서 파일의 내용이 서로 다른 경우에 충돌이 일어날 수 있는데 이 경우에 어떻게 해야 하는가에 대해 이야기해볼 것입니다.

$ git add master.txt
$ git commit -m "6"
$ git branch exp
$ git checkout exp
$ vim exp.txt
$ git add exp.txt
$ git commit -m "7"
$ git checkout master
$ git merge exp
# 정상적으로 merge commit이 생성됨
$ ls -al #exp branch의 파일이 병합됨 확인

파일이 다르면 위의 경우처럼 자동으로 합쳐줍니다.

자동으로 합쳐버린 것이 문제가 된 적이 저는 한 번도 없습니다.

  • stash

다른 브랜치로 checkout을 해야 하는데 아직 현재 브랜치에서 작업이 끝나지 않은 경우는 커밋을 하기가 애매합니다. 이런 경우 stash를 이용하면 작업중이던 파일을 임시로 저장해두고 현재 브랜치의 상태를 마지막 커밋의 상태로 초기화 할 수 있습니다. 그 후에 다른 브랜치로 이동하고 작업을 끝낸 후에 작업 중이던 브랜치로 복귀한 후 이전에 작업하던 내용을 복원할 수 있습니다. 여기서는 이 기능에 대해서 알아봅니다.

stash : 감추다, 숨겨두다

여러분이 작업했던 내용을 어딘가에 숨겨둘 수 있습니다. 그리고 HEAD의 버전으로 이동해서 현재 branch의 상태를 깔끔하게 만들고 다른 branch로 chaeckout할 수 있다 라는 의미입니다.

여러분이 branch를 활발하게 사용하지 않는다면 당장 공부하실 필요는 없습니다.

하지만 어느 순간 stash가 없으면 매우 곤란한 상황이 꼭 생깁니다.

$ vim f1.txt 
$ git add f1.txt
$ git commit -m "1"
$ git checkout -b exp
$ vim f1.txt #내용 수정 중...

수정이 끝나지 않아 commit하기는 애매한데 master branch로 checkout해야 하는 상황입니다.

이 상황에서 checkout 하면

$ git checkout master
$ git status

exp 에서 작업중인 내용이 master까지도 영향을 줍니다.

이 문제에 대한 git의 대답이 stash입니다. 

$ git stash --help

$ git stash
# Saved working directroy and index state WIP on exp

WIP : working in process (작업 중)

"작업 중인 변경 사항들이 save되었다"

$ git status
# 아무것도 수정할 내용이 없다고 나옴
$ git checkout master

문제없이 checkout 할 수 있게 됩니다.

stash해놨던 내용을 복구하는 방법은

$ git stash apply
# f1.txt라는 파일이 다시 수정중인 상태로 돌아옴
$ git stash list

stash한 내용(수정중으로 복구했던 내용)을 날려볼게요.

$ git reset --hard HEAD
$ sit status
# nothing to commit이라 나옵니다.

그렇다면 stash로 복원한 내용을 잃어버린걸까요?

아니에요.

$ git stash list #리스트에 남아있습니다.
$ git stash apply

다시 복원됩니다.

우리가 stash한 내용은 명시적으로 삭제하지 않는 이상 항상 살아있다 라는 것입니다.

$ git reset --hard
$ vim f2.txt
$ git add f2.txt
$ git stash
$ git status #commit 할 것이 없다고 나옵니다.
$ git stash list

가장 최신 것이 0번으로 리스트에 뜹니다. 

$ git stash apply #가장 최신 stash를 적용합니다.
$ git stash drop # 가장 최신 stash를 삭제합니다.
$ git stash list

apply는 가장 최신 stash를 적용하지만 그것을 삭제하지는 않습니다.

$ git stash apply; git stash drop

적용과 삭제를 한번에 실행하는 것 입니다

$ git stash pop

적용과 삭제를 한번에 실행하는 간편한 방법입니다.

stash가 apply하고 drop까지 되는 것입니다.

 

stash는 working directroy의 변경 사항을 감춥니다.

add를 하지 않아 untracked상태의 파일은 stash가 적용되지 않습니다.

stash라는 명령은 적어도 버전 관리를 하고 있는 파일에만 적용된다는 것입니다.
